Transaction atomicity here is correct but implicit. On a mid-stream failure (a bad row at 66-69, or resolve/write), the exception unwinds through the BufferedWriter use{}, which calls PGCopyOutputStream.close() -> endCopy(), flushing the partial rows into the open transaction; the load (including the TRUNCATE at line 40) is then undone only because conn.use{} closes the connection without a commit. It works, but it leans on close() behaviour during stack unwinding. Consider an explicit try/commit + catch/rollback, aborting the COPY with copy.cancelCopy() instead of endCopy() flushing partial data, to make the all-or-nothing guarantee intentional.
