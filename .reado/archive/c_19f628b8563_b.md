Grounded on the actual choropleth query (SELECT state_id, count(*) ... GROUP BY state_id over the whole table): a plain B-tree on state_id will NOT accelerate an unfiltered full-table GROUP BY - Postgres seq-scans + hash-aggregates regardless (at best an index-only scan on a vacuumed table). This index earns its keep for single-state filters / drill-downs (WHERE state_id = ?), not the all-states map. At 200k the seq scan is fine; at scale the real lever is a pre-aggregated rollup (summary table or materialized view refreshed after seed), not this index. Worth stating that rationale since the brief asks for query thinking.

For the all-states query the B-tree on state_id isn't the lever — an unfiltered GROUP BY scans the whole table. As the dataset grows I'd materialise the per-state (and per-hour, per-year) aggregates into a rollup refreshed once after the seed: the read becomes ~50 rows, constant regardless of raw volume. Our dataset is a static batch load, so the rollup never goes stale the usual refresh cost doesn't apply here.
