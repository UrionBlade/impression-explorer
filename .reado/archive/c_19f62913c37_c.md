states.name is the only join key between DB aggregates and the map: the choropleth matches each GeoJSON feature to its count by this string. The seed stores properties.level1 (e.g. Alabama), while map.json also carries properties.name (Alabama, US.AL) and a stable properties.id (US_adm1_29). Two things to pin: (a) the frontend must key features by level1, not name/id, or the join silently misses; (b) consider storing the stable GeoJSON feature id/code alongside the display name, so the join does not hinge on a display string whose formatting could drift. Which field is the intended contract?
